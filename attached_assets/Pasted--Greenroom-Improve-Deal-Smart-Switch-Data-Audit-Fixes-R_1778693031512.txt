## SECTION 1 ‚Äî REMOVE: suggestions that are wrong or never trigger

### 1a. REMOVE ‚Äî cell-mean fallback for flat suggestion (Improve Deal)

**Location:** The fallback path in `generateSwitchSuggestion` (and anywhere in Improve Deal) that computes:

```typescript

```

**Why:** The cell mean for vs√ó$1-5K is $2,058. The mean guarantee on those same deals is $1,466. Using cell mean as the flat suggestion overestimates by $591 per show ‚Äî making the switch more expensive than the original deal. The fallback produces the wrong number in the wrong direction.

**Replace with:** The guarantee amount from the specific deal. See Section 2a.

### 1b. REMOVE ‚Äî "Convert to flat" from Improve Deal for vs deals above $5K gross

**Location:** The `convertToFlat` suggestion block in Improve Deal.

**Current (wrong) behaviour:** Improve Deal offers flat conversion for *any* non-flat deal where SGP confidence tier is A or B ‚Äî including vs deals in the $5-15K and $15K+ buckets.

**Why this is wrong:**
- vs√ó$5-15K: the percentage calculation fires on 91% of shows. A flat at cell mean has an average error of $1,687 per show and a maximum error of $6,038 on a single show.
- The cell earns Tier A (n=102) but individual show prediction is unreliable ‚Äî gross drives payout, not the average.

**What to remove:**

```typescript

//    deal.dealType === 'vs' AND classifyBucket(deal) !== '1_5k'
//    deal.dealType === 'percentage_of_net' AND classifyBucket(deal) !== '1_5k'
//
// These two types above $5K stay as Improve Deal (caps only).
// The flat conversion only belongs to Smart Switch, which already hard-limits at $5K.
```

After this change, `convertToFlat` in Improve Deal only fires for:
- `percentage_of_gross` deals (any bucket) ‚Äî percentage structure is already clean; flat is a cosmetic simplification
- Any deal where Smart Switch does *not* apply and confidence is A/B ‚Äî i.e. not vs or % of net above $5K

## SECTION 2 ‚Äî CORRECT: fix partially right suggestions

### 2a. CORRECT ‚Äî flat suggestion source: use deal.guaranteeAmount, not cell mean

**Applies to:** Smart Switch flat suggestion for `vs` deals in $1-5K bucket.

**The data:** 43 vs deals under $5K, 100% paid the guarantee ‚Äî the percentage calculation never fired once. The guarantee IS the flat answer. The SGP 7-step engine confirms this. The fallback must not use cell mean.

**Implementation:**

```typescript
// In generateSwitchSuggestion, for deal.dealType === 'vs' AND bucket === '1_5k':

function getSuggestedFlat(deal: Deal, sgpResult: SGPResult | null): FlatSuggestion {

  // Preferred path: SGP 7-step engine ran successfully
  if (sgpResult && sgpResult.confidenceTier !== null) {
    return {
      suggestedFlat:        sgpResult.suggestedPrice,
      suggestedFlatRounded: roundTo50(sgpResult.suggestedPrice),
      source:               'sgp_engine',
      confidenceTier:       sgpResult.confidenceTier,
    };
  }

  // ‚úÖ CORRECT fallback for vs deals:
  // The guarantee is the flat ‚Äî verified by 43/43 historical shows.
  if (deal.dealType === 'vs' && deal.guaranteeAmount > 0) {
    return {
      suggestedFlat:        deal.guaranteeAmount,
      suggestedFlatRounded: roundTo50(deal.guaranteeAmount),
      source:               'guarantee_amount',  // document this clearly
      confidenceTier:       'A',
      // Band width: guarantee is exact, but round-to-50 creates ¬±$25 rounding band
      bandLow:  roundTo50(deal.guaranteeAmount) - 25,
      bandHigh: roundTo50(deal.guaranteeAmount) + 25,
    };
  }

  // ‚úÖ CORRECT fallback for percentage_of_net deals in $1-5K:
  // Use the deal's own percentage applied to expected gross with historical expense estimate.
  // Do NOT use cell.avgPayout ‚Äî it blends shows with different percentage rates.
  if (deal.dealType === 'percentage_of_net') {
    const expEstimate = resolveExpenseEstimate(deal.artistId, deal.venueId);
    const cappedExp   = Math.min(expEstimate, SWITCH_CONSTANTS.expenseCap.pn_under5k);
    const netBase     = Math.max(0, deal.expectedGross * 0.9 - cappedExp);
    const flatEst     = netBase * deal.percentage;
    return {
      suggestedFlat:        flatEst,
      suggestedFlatRounded: roundTo50(flatEst),
      source:               'historical_expense_estimate',
      confidenceTier:       expEstimate.tier,  // A if artist history, C/D if genre proxy
      bandLow:  roundTo50(flatEst) - expEstimate.bandWidth,
      bandHigh: roundTo50(flatEst) + expEstimate.bandWidth,
    };
  }

  // ‚ùå No longer fall through to cell.avgPayout. Return null instead.
  return null;  // Mariana sees no suggestion rather than a wrong one
}
```

### 2d. CORRECT ‚Äî confidence tier display: show range when band is wide

**Current wrong behaviour:** Any cell with n‚â•20 earns Tier A and displays a single number. But the vs√ó$1-5K cell (n=43, Tier A) has a p10-to-p90 payout spread of **$1,600** ($1,097 to $2,698). Displaying this as a single flat suggestion is misleading.

**Rule to add:**

```typescript
// After computing confidenceTier from cell.n, apply the band-width override:
function adjustTierForBandWidth(
  tier: 'A' | 'B' | 'C' | 'D',
  bandWidth: number   // p90 - p10 of cell payouts
): 'A' | 'B' | 'C' | 'D' {
  // If the band is wider than $500, the suggestion is uncertain regardless of cell size.
  // Demote display tier so the UI shows a range instead of a single number.
  if (bandWidth > 1000) return tier === 'A' ? 'B' : tier;  // Tier A ‚Üí B: show range ¬±$75
  if (bandWidth > 500)  return tier === 'A' ? 'B' : tier;
  return tier;  // band is tight: keep original tier, show single number
}

// For vs/$1-5K specifically: band = $1,600 ‚Üí always demote to Tier B display
// (Guarantee-based suggestion is still Tier A confidence, but the cell mean fallback is Tier B)
// This distinction matters: if source === 'guarantee_amount', stay Tier A.
// If source === 'cell_mean' (which we've now removed), it would be Tier B.
```

**UI impact:** Tier B shows `$X ¬± $75` instead of `$X`. This is more honest and reduces agent pushback when the actual number differs.

### 2e. CORRECT ‚Äî door hybrid: explicitly handle dead-pool shows and the $15K+ edge case

**Dead-pool shows (9 of 29 door deals):** When `gross √ó 0.9 ‚àí expenseCap ‚â§ 500`, the available pool doesn't exceed the floor. The current formula is silent about this.

```typescript
// In simulateHybridDoor:

export function simulateHybridDoor(
  expectedGross: number,
  expectedExpenses: number,
  floor  = 500,
  split  = 0.60,
  cap    = 1500
): DoorHybridResult {

  const expInput   = Math.min(expectedExpenses, cap);
  const avail      = expectedGross * 0.9 - expInput;

  // ‚úÖ NEW: explicit dead-pool branch
  if (avail <= floor) {
    return {
      artist:        floor,
      venueShare:    0,
      venueNet:      Math.round(avail - floor),  // negative ‚Äî venue absorbs loss
      insuranceGap:  Math.max(0, expectedExpenses - expInput),
      isDeadPool:    true,
      // UI should show: "Artist guaranteed $500 floor. Venue absorbs remaining loss.
      //                  This show needs a minimum gross of ~$2,500 to be viable for both parties."
    };
  }

  // Normal hybrid path
  const above      = avail - floor;
  const artist     = Math.round(floor + split * above);
  const venueShare = Math.round((1 - split) * above);
  const venueNet   = venueShare - Math.round(expectedExpenses - expInput);  // absorb overage
  const insGap     = Math.max(0, (expectedExpenses - expInput) - venueShare);

  return {
    artist, venueShare, venueNet, insuranceGap: insGap,
    isDeadPool: false,
  };
}
```
