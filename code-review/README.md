# Greenroom · Code-Review Bundle

This folder is a **self-contained snapshot of the live Greenroom dataset**
plus a guide that lets a code-review agent (Claude Code, etc.) understand
the prototype **without running the application**.

It pairs the JSON dumps in `data/` with pointers into the real source code
(paths in this repo). A reviewer should be able to open this folder, read
the data shapes here, and then jump straight to the calculation that
produced each field.

> Greenroom is a booker-facing dashboard for a small live-music venue. It
> tracks shows, deals, and settlements; flags follow-ups; and surfaces
> qualitative friction patterns across past deals using an LLM enrichment
> pipeline. The full product narrative and the engineering reference live
> in `exports/greenroom-settlement-redesign-v2.pdf` and
> `exports/greenroom-supporting-document.pdf` at the repo root.

## What's in `data/`

Generated by `artifacts/api-server/scripts/exportCodeReviewBundle.ts`. Re-run
with:

```bash
pnpm --filter @workspace/api-server exec tsx scripts/exportCodeReviewBundle.ts
```

| File | Source query | Rows in this snapshot | What it represents |
|---|---|---:|---|
| `data/shows.json` | `getAllShows()` in `artifacts/api-server/src/lib/queries.ts` | 564 | One row per show in the venue's history (past, today, and the demo upcoming shows). Each row is the fully-joined view used by the **Shows** tab and the Settle wizard. |
| `data/artists.json` | `getAllArtists()` in `artifacts/api-server/src/lib/queries.ts` | 79 | One row per artist with show count, last show date, top deal type, the LLM-derived review-tone topics, and a flagged-attention badge. Powers the **Artists** tab. |

## How the data fits together (schema map)

Authoritative schema: `artifacts/api-server/src/db/schema.ts`. The diagram
below shows only the relationships exercised by this bundle.

```
agencies (id, name)
   ▲
   │ agencyId
agents (id, name, email, agencyId)
   ▲
   │ agentId
artists (id, name, genre, priorShowCount, agentId)
   ▲
   │ artistId
shows (id, venueId, artistId, date, status, ...)
   ▲                        ▲
   │ 1:1 showId             │ showId
deals (showId, dealType,   settlements (showId, status,
   guaranteeAmount,           grossBoxOffice, totalToArtist,
   percentage,                totalExpenses, recoupsJson,
   percentageBasis,           notes, signoffText,
   expenseCap, ...)           positiveSummary, negativeSummary)
                                            │
                                            └─ recoupsJson is a JSON array of
                                               { id, category, label, amount,
                                                 status: "agreed"|"disputed"|"withdrawn" }
```

Other tables in the schema (`ticket_sales`, `comps`, `expenses`,
`switch_suggestions`, `guarantee_suggestions`, `users`, `venues`,
`settings`) are NOT exported here. They are documented in
`exports/greenroom-supporting-document.pdf` and read directly from
`artifacts/api-server/src/db/schema.ts`. The bundle focuses on the three
surfaces the user asked to review: shows, artists, and per-deal insights.

## Per-file shape

### `data/shows.json` — array, one entry per show

```jsonc
{
  "show":     { /* shows row from schema */ },
  "artist":   { /* artists row */ },
  "agent":    { /* agents row, or null */ },
  "deal":     { /* deals row, 1:1 with show */ },
  "settlement": { /* settlements row, or null if not settled */ },

  // Derived flags computed in getAllShows():
  "isUnsupportedDeal":   true,   // dealType ∈ {percentage_of_net, vs, door}
  "isDisputed":          false,  // settlement.status === "disputed" OR any disputed recoup
  "tense":               "past", // "past" | "today" | "upcoming"
  "switchStatus":        null,   // "suggested" | "accepted" | "declined" | null
  "guaranteeSuggestion": null,   // joined row from guarantee_suggestions, or null
  "expenseCategories":           ["production", "hospitality"],
  "recoupCategories":            ["marketing"],
  "disputedRecoupCategories":    []
}
```

The full computation of every derived field is in `getAllShows()` at
`artifacts/api-server/src/lib/queries.ts:42`.

### `data/artists.json` — array, one entry per artist

```jsonc
{
  "artist":  { /* artists row */ },
  "agent":   { /* agents row, or null */ },
  "agency":  { /* agencies row, or null */ },
  "showCount":      18,
  "lastShowDate":   "2026-07-03",
  "topDealType":    "flat",
  "dealTypes": [
    { "dealType": "flat", "count": 8 },
    { "dealType": "percentage_of_net", "count": 6 }
  ],
  "topPositive":    "Clean settlement, paid in full…",   // first sentence of most recent positiveSummary, ~90 char cap
  "topNegative":    "Backline overage remains disputed…",// same for negativeSummary
  "attentionCount": 2                                    // count of getNeedsAttention() items for this artist
}
```

Computed in `getAllArtists()` at
`artifacts/api-server/src/lib/queries.ts:161`.

### `data/insights-per-deal.json` — array, one entry per settled show

This is the bundle the "code review" task was specifically asked for —
**every settled show with its deal context, raw freetext, and the LLM
extracted insight**.

```jsonc
{
  "settlementId":      "stl_show_0000",
  "showId":            "show_0000",
  "showDate":          "2026-04-04",
  "showStatus":        "settled",
  "settlementStatus":  "paid",
  "artist":  { "id": "...", "name": "Low Rooms", "genre": "indie rock" },
  "agent":   { "id": "...", "name": "Cass Burke", "agency": "Independent" },
  "deal": {
    "id": "deal_show_0000",
    "dealType": "flat",         // flat | percentage_of_gross | percentage_of_net | vs | door
    "guaranteeAmount": 2332,
    "percentage": null,
    "percentageBasis": null,    // "gross" | "net" | null
    "expenseCap": null,
    "hospitalityCap": null,
    "bonuses": null,            // parsed from bonusesJson, schema-free shape
    "notes": "Flat $2,332. No upside."
  },
  "settlementFinancials": {
    "grossBoxOffice": 5810,
    "totalToArtist":  2332,
    "totalExpenses":  1707,
    "recoups": [
      { "id": "...", "category": "marketing",          "label": "Co-op marketing spend",
        "amount": 301, "status": "agreed" },
      { "id": "...", "category": "production_overage", "label": "Backline: drum riser",
        "amount": 219, "status": "disputed" }
    ]
  },
  "rawText": {
    "notes":       null,
    "signoffText": "👍"
  },
  "extractedInsight": {
    "positiveSummary": "Clean settlement with artist paid in full at $2,332 guarantee against $5,810 gross box office. Signoff received and marked paid.",
    "negativeSummary": "Backline production overage of $219 for drum riser remains disputed despite settlement completion, creating potential friction.",
    "hasSummary": true
  }
}
```

The two LLM columns (`positiveSummary`, `negativeSummary`) are produced by
`enrichSettlements()` in `artifacts/api-server/src/lib/insights.ts`. The
prompt is gated on the active provider/model from the `settings` table.
523 / 558 settlements in this snapshot have summaries; the remaining 35 are
either rows with no notes/signoff/disputes (so the pipeline intentionally
skips them) or rows that failed enrichment.

## Where each tab gets its data

This map lets a reviewer trace any UI screenshot back to the calculation
that produced it.

| Tab | Frontend page | Backend query | Bundle file that mirrors the data |
|---|---|---|---|
| Shows | `artifacts/greenroom/src/pages/shows.tsx` | `getAllShows` | `data/shows.json` |
| Show detail | `artifacts/greenroom/src/pages/show-detail.tsx` | `getShowById` | one row of `data/shows.json` (super-set of the join) |
| Artists | `artifacts/greenroom/src/pages/artists.tsx` | `getAllArtists` | `data/artists.json` |
| Reports | `artifacts/greenroom/src/pages/reports.tsx` | `getReports` | not in bundle — derive from `data/shows.json` if needed |
| Deal Analysis | `artifacts/greenroom/src/pages/deal-analysis.tsx` | `getDealAnalysis` | not in bundle — derive from `data/shows.json` |
| Needs Attention | `artifacts/greenroom/src/pages/needs-attention.tsx` | `getNeedsAttention` | not in bundle — derive from `data/shows.json` + recoups |
| Insights | `artifacts/greenroom/src/pages/insights.tsx` | `getInsights` + `enrichSettlements` | `data/insights-per-deal.json` (the per-deal LLM output) |
| Settings | `artifacts/greenroom/src/pages/settings.tsx` | `getLlmStatus` / `saveLlmSettings` | n/a (no business data) |

Reports, Deal Analysis, and Needs Attention are all *roll-ups over the
same underlying joins already exposed in `data/shows.json`*, so a
reviewer doesn't lose anything by not having those JSON files separately —
the calculation rules are documented in `replit.md` (root) and in
`exports/greenroom-supporting-document.pdf`.

## How to do a code review of the prototype with this bundle

1. **Read `replit.md` at the repo root first.** It is the canonical
   product overview: every tab, every API endpoint, every calculation, and
   the architecture decisions that govern the codebase.
2. **Read `exports/greenroom-supporting-document.pdf`.** Mirrors the
   replit.md material in a presentation-ready form with cross-references
   to the redesign argument.
3. **Open `data/shows.json` and `data/artists.json`** to see the actual
   shapes the frontend consumes. Compare them to the type definitions in
   `artifacts/greenroom/src/lib/types.ts`.
4. **Open `data/insights-per-deal.json`** to inspect the LLM-extracted
   insight per deal. Sanity check the `positiveSummary` /
   `negativeSummary` fields against the raw `notes` / `signoffText` and
   the disputed recoup lines in the same row.
5. **Trace any field back to its calculation:**
   - All non-LLM derived fields live in
     `artifacts/api-server/src/lib/queries.ts` (one function per tab).
   - LLM-derived fields live in
     `artifacts/api-server/src/lib/insights.ts` (per-settlement summary)
     and `artifacts/api-server/src/lib/showExport.ts` (per-show
     deep-summary used by `GET /api/shows/:id/export`).
   - The single LLM helper is `artifacts/api-server/src/lib/llm.ts` —
     swapping providers is a one-line setting change.
6. **HTTP routes are in `artifacts/api-server/src/routes/greenroom.ts`.**
   The full list is in `replit.md` under "API surface".

## Reproducing the snapshot

```bash
# Re-export the JSON in this folder against the current DB
pnpm --filter @workspace/api-server exec tsx scripts/exportCodeReviewBundle.ts
```

The script writes to `code-review/data/` (this folder). It is idempotent
and safe to re-run after demo seed updates.

If you need to rebuild the underlying database from upstream first, see
the recipe in `artifacts/api-server/data/seeds/README.md`.
